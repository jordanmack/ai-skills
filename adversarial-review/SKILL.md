# Adversarial Review

Drive a work artifact — a **plan** or an **implementation** — through repeated adversarial review by other models until a round produces no serious finding that survives your adjudication. You, the current host model, are **not** a reviewer. You are the **adjudicator**: you decide which findings are real, fix the ones worth fixing now, maintain a **skip list** of already-dismissed issues, and run the loop.

The artifact is the plan, code, file, or diff in play. Don't ask the user to clarify the artifact's *content* — infer it. The one question worth asking about the artifact is whether the target is a plan or an implementation, and only when you genuinely can't tell. (Roster gaps are a separate, allowed question — see below.)

## Dispatching reviewers

Route each reviewer by whether the current harness can run it natively:

- **In-harness models → subagents.** A model the harness can spawn directly (e.g. in Claude Code, Claude models like Opus and Sonnet) should be launched as a subagent. Infer which models are native — don't assume a fixed mapping, since harnesses and lineups change.
- **External models → shell.** A model the harness can't spawn natively (e.g. Grok or Codex from inside Claude Code) is reached externally, typically by shelling out. Prefer this repo's cross-tool second-opinion skills for the exact command.

If a named reviewer cannot be reached — no subagent path and no working shell command — **hard-fail the whole loop** and tell the operator which reviewer was unreachable. "Reached" means a genuine review actually came back — this covers the whole dispatch-through-collect window, not just the initial launch. A reviewer that crashed, or returned an auth error, rate-limit notice, or empty/garbage output produced *no review* — treat it as unreachable and hard-fail, never as "found nothing serious." (A coherent *content refusal* — the reviewer declines the artifact on policy grounds — is the one no-review case that is **not** a hard-fail; it has its own disposition below.)

Two distinctions matter here, because both can hard-fail a loop that should keep running:

- **Empty output ≠ "found nothing."** A clean review must *affirmatively* say so — that's why step 1 asks each reviewer to state plainly when it found nothing serious. Treat a positive "nothing serious" as a valid, complete review (it counts toward a clean round). Treat genuinely empty, whitespace-only, or garbled output as no-review → hard-fail. If you can't tell which you got, re-dispatch that one reviewer once to disambiguate before hard-failing — don't guess.
- **A timeout is retryable before it is fatal.** A single timeout is not an immediate whole-loop hard-fail: retry that reviewer (optionally scoped per step 1 — instruct it to return after a fixed number of serious issues) before giving up. Only when a reviewer *repeatedly* fails to return a review across retries is it unreachable → hard-fail. This is what makes step 1's "repeatedly hits timeouts → scope it" remedy reachable; a one-strike rule would render it dead.
- **A content refusal is not an infrastructure failure — don't loop on it forever.** A reviewer that *can* be reached but declines the artifact on policy grounds (e.g. it won't review security-tooling, exploit, or otherwise sensitive code) will refuse every round identically; treating that as "unreachable → hard-fail" deadlocks the loop with no exit and mislabels a principled refusal as an outage. Instead, surface it accurately to the operator as a **content refusal by reviewer X**, and offer roster repair: drop X (with the operator's consent) and continue with the rest, or substitute another model. Proceeding with a deliberately repaired roster is allowed; silently faking X's review is never. If *every* reviewer refuses, stop and report — the artifact can't be reviewed by this roster. **Minimum-roster floor:** repair must leave **at least one** genuine reviewer actually reviewing — an independent cold subagent counts, including the host model when the operator explicitly named it (see *The roster*). If refusals, drops, and host-exclusion would collapse the roster to zero — *or* to a single model when the operator asked for cross-model review — stop and tell the operator the roster has degenerated rather than certifying a "consensus" that one model (or none) produced. The skill's whole premise is independent cross-model angles; a one-model exit is not that, and must be the operator's explicit choice, not a silent consequence of repair.
  - **Classifying it:** a *coherent statement declining the task* is a refusal (→ roster repair); *empty, whitespace, or garbled* output is no-review (→ hard-fail). These have opposite consequences, so when output is ambiguous — a refusal that got truncated mid-sentence, say — re-dispatch that one reviewer once to disambiguate before deciding, the same one-shot retry the empty-vs-nothing-found case gets. Don't guess between buckets when a cheap retry settles it.

A partial roster breaks the exit criterion: a clean round is supposed to mean *every reviewer on the active roster* (as repaired, if repair happened) swept the artifact and you cleared what they raised, so a silently-missing reviewer means the artifact was never reviewed from that model's angle — exit would rest on a sweep that never happened. (Exit is keyed on your adjudication, not on reviewer unanimity — but it still presumes the full roster actually reviewed.) Never fake a review by role-playing the missing model.

## The roster

Parse the reviewer names from the free-text invocation argument (e.g. `opus, grok` or `sonnet and grok`).

- If "Claude" is named generically, use a non-host Claude model; if more than one fits, or the host isn't a Claude model, ask which.
- **The host model is a reviewer only when the operator named it explicitly.** By default the host doesn't review its own work — a model grading itself inline is not the independent angle this skill exists for. But if the operator *deliberately named the host model* in the roster (e.g. the host is Opus and the arg says `opus`), honor it: dispatch it as a **separate, cold subagent** (its own fresh context, the artifact inlined like any reviewer), never as the inline host session adjudicating itself. That preserves independence — a second instance of the same model attacking the artifact blind — while respecting the operator's explicit choice. Tell the operator you're running the host model as a cold-subagent reviewer on their say-so. A merely *generic* mention that happens to resolve to the host (bare "Claude" on an Opus host) is **not** an explicit naming: drop it per the line above and say so. When in genuine doubt whether a name targets the host on purpose, ask.
- If no models are named — or the roster ends up empty — ask who the adversaries should be. Don't invent a default roster.

## The loop

1. **Dispatch.** Send the current artifact to every reviewer, independently, asking each for an adversarial review — find the flaws, the holes, the risks. Reviewers start cold, so make each brief self-contained: the artifact itself (inline it for external/shell reviewers, who can't resolve a workspace path; a pointer is fine only for subagents that share your workspace), the goal it must satisfy, and the **serious/trivial definitions from *Exit criteria*, quoted** so they label findings the way you'll adjudicate them. Ask each to say plainly if it found nothing serious. Apply a **suggested timeout of 30 minutes per reviewer agent**; adjust up or down based on the size and complexity of the work. If a reviewer repeatedly hits timeouts, consider scoping its work — for example, instructing it to return immediately after finding a fixed number of serious issues (e.g. 3–5). This is safe *mid-loop* because the loop is iterative: any issues not surfaced in one round will be found in subsequent rounds. ⚠️ **A scoped reviewer can certify the exit round only on an *explicit* completion statement — never on mere absence of findings.** An output cap ("return after 3 serious issues") truncates a review only when that many issues are found, so a scoped reviewer that finds nothing *might* have swept the whole artifact — but "might have" is a guess, and a guess can't gate exit. Require each scoped reviewer to state plainly whether it **completed the full sweep** or **stopped at the cap** (or ran out of room). Only an **output cap** (stop after N findings) is compatible with certifying a clean round; *narrowing the artifact surface* a reviewer sees (by area, or "just the changed parts") can never certify exit, because it never sweeps the whole artifact — use surface-narrowing only as a mid-loop accelerant, never on the round you exit on. A round is clean only if every contributing reviewer either ran un-scoped or *explicitly affirmed a completed full-surface sweep*. Treat a bare "nothing serious" with no completion claim the same as a cap-hit: not a clean signal. **The transition when this gates a would-be-clean round is defined, not a spin:** re-dispatch that one reviewer un-scoped (or scoped-but-asked to confirm full-surface completion) and use *that* result to decide cleanliness. The round can't be declared clean on the ambiguous response, and it can't loop unchanged either — the un-scoped re-dispatch is the forced next step. And because even an explicit completion claim is an unverifiable self-report (a reviewer that exhausted its context mid-artifact may still claim "done"), if exit rests on a single scoped-but-completed certification and you have any doubt, re-dispatch that reviewer un-scoped to confirm — the same disambiguation the empty-output case gets. The iteration guarantee that makes scoping safe dies on the exit round; don't let an unverified completion claim be the only thing holding it up.

   Keep reviewers **cold every round**: send only the updated artifact plus the skip list (see *The skip list*). Never pass your adjudication, rejections, or prior findings back **as free-text commentary** — feeding them your verdicts conversationally manufactures false consensus, and their independence is the whole point. A finding you didn't fix by editing the artifact — rejected, already-settled, or skipped — will likely recur every round; that's expected and doesn't block exit, since exit is keyed on your adjudication, not reviewer silence. If the recurrence bothers you, the sanctioned responses are two: make the artifact **self-defending** (write the rationale *into the work itself* — a code comment, a note in the plan — content that stands on its own merits for any reader; note this is an artifact edit, so it re-dispatches like any fix), or list the specific *dismissed* finding (rejected or already-settled **only** — never a real Defer/Skip finding) on the **skip list** so reviewers don't re-spend a round rediscovering it. Never annotate the dispatch with loose "I rejected this" prose or argue back against a reviewer's finding. There is a second, narrow exception to coldness beyond the skip list: you may supply a *fact a reviewer structurally couldn't know* (e.g. "tests live elsewhere"); never your judgments.

   **The skip list is a deliberate, bounded exception to the cold rule.** It is *not* free-text feedback and *not* a channel for your reasoning about live findings. It carries only already-adjudicated, issue-specific dismissals, formatted as the spec below requires. Everything outside the skip list stays cold.
2. **Collect** every reviewer's findings.
3. **Refute.** Consolidate every reviewer's raw findings into one deduplicated list of unique findings (if it is empty, skip this pass), then send that consolidated list back to each reviewer and ask it to try to *refute* every finding on it — its own and the others' alike — before the findings reach you. Each reviewer returns, per finding, whether it survives refutation or falls, with reasoning. This is a cross-examination pass: a finding one model raised must withstand the whole roster trying to knock it down. The refutation verdicts are *input to your adjudication*, not a substitute for it — you still decide. See *The refutation pass*.
4. **Adjudicate** each finding (rules below), informed by the refutation verdicts. Don't apply findings blindly — and don't drop a finding solely because reviewers refuted it; weigh their refutation on its merits the same way you weigh the original finding.
5. **Fix** the findings you classified as real and worth-doing-now.
6. **Update the skip list** (see *The skip list*): re-validate existing entries against the now-current artifact and drop any that no longer hold, then append this round's new false-positive and already-settled dismissals.
7. **Check the exit test** (see *Exit criteria*) before looping: if this round was clean *and* fix-free, exit. If you applied any fix, you must re-dispatch to verify it.
8. **Repeat** from step 1 with the updated artifact and skip list, or **exit and report** per step 7.

## The refutation pass

Between collecting findings (step 2) and adjudicating them (step 4), the roster cross-examines itself. The goal is to kill plausible-but-wrong findings *before* they cost you adjudication effort, and to harden the survivors with the roster's best counter-arguments already on record.

**Consolidate first.** Merge all reviewers' raw findings into a single list of *unique* findings — deduplicate near-identical reports (same defect described two ways collapses to one entry), and strip the attribution so no reviewer can tell which finding was its own. Number the entries; you'll key the returned verdicts to these numbers. Findings already on the **skip list** don't enter the refutation pass — they were dismissed in a prior round and the skip list already handles their suppression. A carve-out re-raise (a listed issue raised anew with a genuinely different argument) is a live finding and does enter.

**Dispatch the refutation.** Send the consolidated list to *every* reviewer in the roster — the same reviewers, kept just as cold as in step 1 (artifact + skip list + the consolidated list; never your own leanings about which findings are real). Ask each, for every numbered finding, to argue the *strongest case that it is wrong* — false positive, already mitigated, misreads the artifact, or too trivial to matter against the quoted serious/trivial definitions — and then state a verdict: **survives** or **refuted**, with reasoning. A reviewer refuting its own finding is expected and valuable, not a contradiction. Apply the same timeout, scoping, and unreachable/refusal rules as step 1: a reviewer that can't be reached for the refutation pass is unreachable for the round (retry, then hard-fail per *Dispatching reviewers*) — don't silently skip its refutation.

**Then adjudicate (step 4), using the verdicts as evidence, not as the verdict.** The refutation pass informs your adjudication; it does not replace it. Specifically:

- A finding the roster *unanimously refuted* is a strong signal it's a false positive — but verify the refutation actually holds against the artifact before dismissing, exactly as you'd verify an original finding. A confidently-wrong refutation is as possible as a confidently-wrong finding.
- A finding that *survived* refutation is a hardened finding — the roster tried to kill it and couldn't — so it deserves real weight, but you may still reject it on grounds the reviewers missed (e.g. a fact they structurally couldn't know).
- When reviewers *split* on a finding (some refute, some say it survives), adjudicate the disagreement on its merits — don't count votes. The refutation arguments on both sides are exactly the material you need to decide.

The refutation pass changes *what reaches your adjudication and how well-argued it is*; it does not change the exit criteria, the skip-list rules, or the defer/skip ledger. A finding you dismiss because the roster refuted it is still a rejection you record (and, if appropriate, a skip-list entry with the refutation as its rationale); a finding that survived and that you judge real, serious, and now-worthy still blocks a clean round.

## The skip list

The skip list is a running exclusion file the **adjudicator owns and writes** — it is your serialized adjudication, prepended to every reviewer's prompt to stop the loop re-spending rounds on findings you've already dismissed. It is the primary sanctioned exception to the cold rule (the others are the structurally-couldn't-know fact above and the consolidated findings list sent during the refutation pass), and it earns that exception by being narrow.

**Why it exists:** without it, every dismissed finding recurs each round, and the loop burns reviewer attention (and your re-adjudication effort) re-litigating settled noise instead of hunting new surface. Killing that churn is the point — and it lets the list grow across rounds, by design.

**What goes on it — only these, and only at issue granularity:**

- **False positives** you rejected (the finding doesn't hold against the artifact).
- **Already-settled** findings (decided against earlier, in your context / ADRs / commits / the plan).

**Never** put real-but-deferred or real-but-skipped findings on it — those are real issues you've merely chosen not to fix now, and suppressing them would hide live problems. They go in the final report only.

⚠️ **Do not confuse the *skip list* with the adjudication disposition *Skip*** — they are opposites despite the shared word. The **skip list** holds findings that are *not real* (false positives) or *settled*; putting one there tells reviewers to stop raising it. The adjudication **Skip** (see *Adjudication*) is a finding that *is* real and serious but not worth fixing — it must be **reported to the user** and must **never** touch the skip list. Suppressing a real Skip-disposition finding by listing it would hide a live defect, the exact failure the skip list is forbidden from causing. Mnemonic: the skip list suppresses *non-issues*; a Skip disposition *flags a real issue you chose to live with*.

**Format every entry as a single specific issue plus its dismissal rationale — never an area, never a bare issue.** Two required parts: the specific claim, and *why* you dismissed it. The rationale is mandatory, not decorative: it is the thing a reviewer compares its own reasoning against to decide whether the carve-out applies (raise it only with a *different* argument). An entry with no rationale makes "does my reasoning match?" unanswerable, so the carve-out collapses and the entry becomes blanket suppression. An area-scoped entry silences neighboring issues that happen to live nearby; an issue-scoped entry suppresses only the exact claim you adjudicated.

- ❌ `auth handling — dismissed` (silences every future auth finding)
- ✅ `missing null-check on token at L42 — dismissed: validated upstream in middleware`

**Re-validate and drop each round (step 6).** An entry can go stale two ways: the **artifact** mutates (a finding that was a false positive in round 1 can become *real* in round 3 after you edit nearby code), or the **external state the rationale rests on** changes without any artifact edit (an ADR is added, a context decision reverses, a "settled" call is overturned). Either way, before re-dispatching, recheck each entry against the current artifact *and* its still-standing rationale, and **drop any whose rejection no longer holds** — that finding must be raisable again. (The external-state case is why re-validation is not a no-op on a fix-free round.) An append-only list silently carries a now-wrong "false positive" verdict straight to exit. Issue-specific scoping limits which *neighbors* a stale entry can hide; re-validation handles the entry turning real on *itself*. You need both.

**When a reviewer uses the carve-out, update the entry's rationale.** If a reviewer re-raises a listed issue with a genuinely new argument (as the carve-out permits) and you adjudicate it *real*, drop the entry and put the finding back through normal adjudication — now/defer/skip, same as any real finding (it is not automatically a fix-now; it may be real-but-deferred or real-but-skipped). But if you reject it *again*, you must **rewrite the entry's rationale to defeat the new argument** before the next round — otherwise the reviewer's reasoning still won't "match" the stale rationale, it re-raises the same stronger argument every round, and the loop never converges. The rationale must always reflect your *current* best reason for dismissal, including rebuttals to every argument raised against it.

**Prompt framing.** Introduce the list to reviewers as: *"The following specific issues were already adjudicated and dismissed on the rationale shown — do not re-raise an issue if your reasoning matches its rationale. You MAY raise an issue on this list if you have a genuinely different argument that the stated rationale does not address, and say so explicitly. Otherwise spend your effort finding new issues."* This suppresses rote re-reports while leaving in scope both new problems in the same area and a re-raise of a listed issue that defeats the original rationale — the one channel by which a wrong dismissal can be independently rediscovered. The carve-out is deliberately narrow: matching reasoning is suppressed, a stronger new argument is not.

## Adjudication

When reviewing your own work, resist defensiveness — weigh each finding on its merits as if someone else wrote it. For every finding, decide with reasoning:

1. **Real or false positive?** Verify it actually holds against this artifact; a reviewer without full context will raise things that don't apply. Reject false positives explicitly — don't drop them silently.
2. **Already settled?** Check whether this was decided against — **first in your own context** (the live conversation often holds the decision and its reasoning), then ADRs, comments, commits, the plan. Re-litigating a settled decision is not actionable. But *absence* of a record is not a rejection: a fresh plan has no ADRs or history, so treat "nothing on record" as not-yet-settled and judge on merits.
3. **Serious or trivial?** Classify by *your own* judgment using the *Exit criteria* definitions — **not** the reviewer's label. Exit is keyed on adjudicated severity, so a real correctness/security bug a reviewer mislabeled "trivial" is still serious, and a reviewer's "serious"-labeled nit is still trivial. Get this wrong and a real bug slips out as non-blocking.
4. **Now, defer, or skip?** For findings that are real and not settled:
   - **Now** — fix it this round.
   - **Defer** — real, but out of scope for this pass.
   - **Skip** — real but not worth doing (cost outweighs value, edge case that won't occur).

   Defer and skip findings must **not** be written into the artifact or the skip list, and must **not** be fixed — but you must **track them in the same append-only ledger** the skip list uses (a separate section of it), because they are *real* defects and losing them across a long loop or a context compaction hides live problems — a strictly worse failure than losing a dismissed non-issue. Record each with its rationale when you first defer/skip it; they surface to the user in the final report and nowhere else.

When reviewers disagree, adjudicate the disagreement — don't average it. One reviewer's objection can be valid even if the others missed it.

## Exit criteria

A round is **clean** when **no serious finding is left to fix now** — nothing you adjudicated as real, serious, and now-worthy. This is keyed on your adjudicated verdict, not the reviewers' raw labels: if every reviewer raised serious-looking findings but you correctly rejected them all, that round is clean. Serious findings you *deferred* or *skipped* don't break cleanliness either (you've consciously chosen not to fix them now) — but they must appear in the final report. A round in which every reviewer ran a *complete review* and surfaced no findings (or only trivial ones) is also clean — but note this means **reviewers affirmatively reviewed and reported nothing serious**, not that their output was empty. Empty/garbled output is no-review → hard-fail (see *Dispatching reviewers*), never a clean round.

The loop ends on **one clean round**. The skip list stays attached on the exit round (no list-free final pass): a cold finale would re-flood you with exactly the dismissed not-issues the list exists to suppress, pulling focus off real new findings for no gain. The residual risk is explicit and accepted — a finding you adjudicated wrong stays suppressed to exit — and it is bounded by adjudication quality alone, which (since the adjudicator owns the skip list) is already the system's trust root. **One corollary of one-round exit, stated plainly so it isn't mistaken for full coverage:** dismissals you *mint on the exit round itself* (false positives you reject, entries you append at step 6 of the final round) are never re-presented to a later cold round, so the carve-out — the channel by which a wrong dismissal gets independently rediscovered — structurally cannot fire for them. The carve-out backstops dismissals from *earlier* rounds; exit-round dismissals rest on adjudication quality alone. Adjudicate the final round's rejections with extra care for exactly this reason.

Consensus requires a **fix-free** clean round: if you edited the artifact this round, you have not verified the result, so you must re-dispatch — a round in which you applied any fix is never the clean round. (Re-validating or pruning the skip list does not count as editing the artifact.)

**A re-opened entry blocks exit.** Step 6's re-validation can flip a dismissed entry back to *real* on the very round you were about to exit on. If it does, that finding has never faced a live reviewer since re-opening — so the round is **not** clean, regardless of whether you fix, defer, or skip the re-opened issue. Drop it from the skip list and re-dispatch at least once more so reviewers see the now-live artifact surface cold. Exit only on a round where re-validation re-opened nothing.

⚠️ **A first-appearance dismissal minted on the exit round gets neither backstop — adjudicate it with extra care.** Step 6 re-validates *existing* entries and *then* appends this round's new dismissals, so a finding raised for the first time on the would-be-exit round and rejected as a false positive is appended *after* re-validation runs — it is never re-validated, and (per the exit-round note above) the carve-out can't re-present it either. The re-open guard below is scoped to existing entries flipping real; it does not reach a fresh append. There is no structural catch for a wrong fresh exit-round dismissal, so it rests on adjudication quality alone — treat every rejection you make on a clean-looking round as the highest-stakes call in the loop, because it is the one with no second look.

⚠️ **Re-open only on a genuine change — don't ping-pong.** Re-validation may re-open an entry only when the artifact *or* the external state its rationale rested on has **actually changed** since you last adjudicated it. An entry sitting near your decision boundary must not re-open on unchanged evidence just because you'd weigh it slightly differently today — that creates a re-open → re-dispatch → re-dismiss → re-open cycle that never terminates (re-validation and adjudication apply the same real-or-false-positive test, so a borderline entry can oscillate forever). Rule: once you re-dispatch a re-opened entry and *re-confirm* the dismissal, that dismissal is **sticky** — record that you re-confirmed it, and do not re-open it again unless the underlying artifact/external state changes anew. Convergence depends on re-opens being driven by new facts, never by adjudicator vacillation. **One exception:** if you recognize your prior rationale was *logically wrong* (not merely a borderline call you'd now weigh differently — an actual error in the reasoning), re-open and correct it; a known-bad dismissal must never be frozen by stickiness. The bar is "the rationale is demonstrably mistaken," not "I'm no longer sure" — the latter is the vacillation stickiness exists to stop.

- **Serious** = correctness, security, data loss, architectural flaws, broken requirements — anything that changes whether the work is right.
- **Trivial** = nits, style, wording, micro-optimizations, preference. These never block exit.

## Final report

On every stop — clean exit, hard-fail, or refusal/roster degeneration — give the user (mark **Outcome** accordingly):

- **Outcome** — how many rounds ran; clean-round reached or still-in-progress.
- **Roster** — the reviewers that actually reviewed, and any roster changes during the run: refusals, operator-consented drops, substitutions, and any reviewer that ran scoped. If the roster was repaired, say so plainly — a "clean exit" means nothing without knowing who did and didn't review.
- **Fixed** — the real issues you addressed, briefly.
- **Deferred** — real issues left for later, with why.
- **Skipped** — real issues judged not worth doing, with why.
- **Rejected** — every *serious* finding you overruled (false positive or already-settled), not just the notable ones — especially any the reviewers raised unanimously. This is where your judgment is most overridable, so make it complete and easy to scan. Note for each whether the refutation pass played a part — a finding the roster unanimously refuted that you then rejected is a different kind of call than one you overruled against the roster's own survival verdict, and the user should be able to tell them apart.
- **Skip list** — every entry that was *ever* on the skip list during the run, *including the trivial ones and the ones you later dropped*. The skip list suppressed these from reviewers, so leaving any off the report (trivial false positives, which the Rejected bullet doesn't require; or entries re-validated away in a later round) would create a suppression channel with no audit trail. Because re-validation removes stale entries from the *live* list each round, keep a separate append-only ledger of entries as you add them — a dropped entry still held a finding back for the rounds it was active, and "during the run" means it must still be reported (mark it as dropped, with when and why). List every entry and its dismissal rationale so the user can see exactly what was held back and why.
